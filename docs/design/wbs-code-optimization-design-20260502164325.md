# WBS 编码优化设计方案

> 版本：1.0
> 日期：2026-05-02
> 状态：待评审

---

## 一、背景与目标

### 1.1 当前问题

- WBS 编码存储在数据库中，增删改操作后需要复杂的重排逻辑
- 不同用户看到不同任务，但编码是全局统一的，无法满足多视角需求
- 编码作为存储属性，导致引用关系复杂，维护成本高

### 1.2 优化目标

| 目标 | 说明 |
|------|------|
| **编码即视图** | WBS 编码是查询时实时计算的显示属性，不持久化存储 |
| **UUID 为锚** | 系统内部所有关联仅通过 UUID |
| **用户视角隔离** | 不同用户根据权限范围看到不同的编码 |
| **实时重排** | 增删改操作后立即重新计算编码 |

---

## 二、核心设计原则

| 原则 | 说明 |
|------|------|
| **编码即视图** | WBS 编码是查询时实时计算的显示属性，不持久化存储 |
| **UUID 为锚** | 系统内部所有关联（前置任务、父子关系）仅通过 UUID |
| **用户视角隔离** | 不同用户根据权限范围看到不同的编码 |
| **实时重排** | 增删改操作后立即重新计算编码 |
| **准确性优先** | 缓存优化不能影响数据准确性 |

## 三、数据模型变更

### 3.1 移除字段

| 字段 | 说明 |
|------|------|
| `wbs_code` | 不再存储，改为实时计算 |
| `wbs_order` | 不再存储，排序由 sort_order 字段支持 |

### 3.2 保留/新增字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID | 唯一标识，系统内部关联使用 |
| `parent_id` | UUID (nullable) | 父任务关联 |
| `wbs_level` | int (1-5) | 层级深度，范围 1-5 |
| `sort_order` | int | 手动排序值，拖拽后存储；为空时按 created_at 排序 |
| `project_id` | UUID | 所属项目 |

### 3.3 数据库迁移

- 现有测试数据可直接删除
- 迁移脚本：删除 `wbs_code` 和 `wbs_order` 字段
- 新增 `sort_order` 字段（如不存在）

## 四、编码规则

### 4.1 编码格式

| 规则 | 说明 | 示例 |
|------|------|------|
| 格式 | 纯数字点分 | `1`, `1.2`, `1.2.3` |
| 最大层级 | 5 级 | `1.2.3.4.5` |
| 第 5 级限制 | 禁止创建子任务 | - |

### 4.2 排序依据

| 优先级 | 字段 | 说明 |
|--------|------|------|
| 1 | `sort_order` | 用户手动拖拽后存储的排序值 |
| 2 | `created_at` | 未手动排序时按创建时间 |

### 4.3 前置任务引用

| 场景 | 处理方式 |
|------|----------|
| 存储 | `predecessor_id` 存储 UUID |
| 显示 | 查询时实时计算并返回 `predecessor_code` |

### 4.4 全局视图编码规则（重要）

**核心原则：WBS 编码在用户可见视图下全局连续递增，不区分项目。**

| 规则 | 说明 |
|------|------|
| **根任务编号** | 所有根任务（WBS等级1）从 1 开始连续递增，不区分项目 |
| **子任务继承** | 子任务继承父任务编码，如父任务编码为 4，则子任务为 4.1, 4.2, 4.3... |
| **视图层面** | 编码在查询时实时计算，基于用户可见的所有任务统一计算 |

**示例：**
```
根任务 1 → MX20台车硬件系统方案设计 (项目2198)
根任务 2 → MX20嵌入式系统设计 (项目2198)
根任务 3 → MX20核心风险 (项目2198)
根任务 4 → 3702项目里程碑 (项目3702)  ← 新项目，编码继续递增
  ├─ 子任务 4.1 → 开发样机 (项目3702)
  ├─ 子任务 4.2 → 注册样机 (项目3702)
  └─ 子任务 4.3 → 工程样机 (项目3702)
根任务 5 → 3702平板导入 (项目3702)
  └─ 子任务 5.1 → 合规线 (项目3702)
根任务 6 → 2188心脏机控制面板固件 (项目2188)  ← 又一个新项目
```

**对比旧方案：**
```
旧方案（按项目独立编号，已废弃）：
项目2198: 1, 2, 3, 4, 5...
项目3702: 1, 2, 3, 4...  ← 每个项目从1重新开始
项目2188: 1, 2, 3...     ← 编号重复，不直观

新方案（全局连续编号）：
所有项目: 1, 2, 3, 4, 5, 6, 7, 8...  ← 全局连续
```

## 五、操作行为

### 5.1 新增任务

| 步骤 | 说明 |
|------|------|
| 1 | 用户指定插入位置（某个任务之后或列表末尾） |
| 2 | 系统分配 UUID，设置 `sort_order` |
| 3 | 失效该项目所有用户的缓存 |
| 4 | 查询时重新计算编码，后续兄弟任务序号自动后移 |

### 5.2 删除任务

| 步骤 | 说明 |
|------|------|
| 1 | 检查该任务是否被其他任务引用为前置任务 |
| 2 | 如有引用，提示用户"该任务被其他任务引用为前置任务，删除后自动解除引用，是否继续？" |
| 3 | 用户确认后，自动解除所有指向该任务的前置任务关系 |
| 4 | 删除任务及其所有子任务（级联删除） |
| 5 | 失效该项目所有用户的缓存 |
| 6 | 查询时重新计算编码，后续兄弟任务序号自动前移 |

### 5.3 拖拽排序

| 步骤 | 说明 |
|------|------|
| 1 | 用户拖拽任务到新位置 |
| 2 | 更新 `sort_order` 值 |
| 3 | 失效该项目所有用户的缓存 |
| 4 | 查询时重新计算编码 |

### 5.4 移动任务（修改父任务）

| 步骤 | 说明 |
|------|------|
| 1 | 用户拖拽任务到新的父任务下 |
| 2 | 检查移动后子任务是否会超过 5 级，如会则禁止操作并提示用户 |
| 3 | 更新 `parent_id` 和 `sort_order` |
| 4 | 重新计算被移动任务的 `wbs_level`（新层级 = 新父任务层级 + 1） |
| 5 | 递归调整所有子任务的 `wbs_level`（相对层级保持不变） |
| 6 | 子任务一并移动，保持相对关系 |
| 7 | 失效该项目所有用户的缓存 |
| 8 | 查询时根据新位置重新计算编码 |

### 5.5 跨项目移动

| 步骤 | 说明 |
|------|------|
| 1 | 用户拖拽任务到另一个项目 |
| 2 | 检查是否有前置任务关系，如有则提示用户是否保留 |
| 3 | 弹出确认框，提示影响范围 |
| 4 | 确认后更新 `project_id`、`parent_id`、`sort_order` |
| 5 | 重新计算被移动任务及其子任务的 `wbs_level` |
| 6 | 失效源项目和目标项目所有用户的缓存 |
| 7 | 查询时重新计算编码 |

### 5.6 复制任务

| 步骤 | 说明 |
|------|------|
| 1 | 用户选择任务并点击复制 |
| 2 | 检查原任务是否有前置任务，如有则提示用户是否保留 |
| 3 | 系统生成新 UUID，复制任务数据 |
| 4 | 插入到选中行之后 |
| 5 | 子任务一并复制，生成新 UUID |
| 6 | 失效该项目所有用户的缓存 |
| 7 | 查询时重新计算编码 |

## 六、导入导出

### 6.1 设计原则

| 原则 | 说明 |
|------|------|
| **所见即所导** | 导出时，用户看到什么就导出什么 |
| **导入即新建** | 导入任务等同于批量新建，系统生成新 UUID |
| **层级定结构** | 根据 `wbs_level` 层级按顺序建立父子关系 |
| **编码重计算** | 导入后按当前用户视角重新计算 WBS 编码 |

### 6.2 导出

| 字段 | 处理方式 |
|------|----------|
| 所有可见字段 | 原样导出，用户看到什么就导出什么 |
| WBS 编码 | 导出当前用户视角的编码值 |
| WBS 层级 | 导出 `wbs_level` |
| 其他字段 | 按现有逻辑导出 |

### 6.3 导入

| 步骤 | 说明 |
|------|------|
| 1 | 解析导入文件，按行顺序读取任务 |
| 2 | 为每个任务生成新 UUID |
| 3 | 根据 `wbs_level` 层级，按导入顺序建立父子关系（见下方算法） |
| 4 | 前置任务关系不导入，需手动重新设置 |
| 5 | 插入任务到目标项目 |
| 6 | 失效目标项目所有用户的缓存 |
| 7 | 查询时按当前用户视角重新计算 WBS 编码 |

### 6.4 导入时建立父子关系算法

```
按导入顺序遍历任务：
1. 维护一个层级栈，记录每个层级最近的父任务
2. 对于当前任务：
   - 层级 = 1：根任务，parent_id = null
   - 层级 > 1：parent_id = 层级栈[层级 - 1]
3. 将当前任务加入层级栈[层级]
4. 继续处理下一个任务

示例：
导入数据（按顺序）：
  任务A, 层级=1 → parent_id=null, 栈[1]=A
  任务B, 层级=2 → parent_id=A, 栈[2]=B
  任务C, 层级=3 → parent_id=B, 栈[3]=C
  任务D, 层级=2 → parent_id=A, 栈[2]=D（覆盖B）
  任务E, 层级=3 → parent_id=D, 栈[3]=E（覆盖C）
  任务F, 层级=1 → parent_id=null, 栈[1]=F（覆盖A）

最终结构：
  A
  ├─ B
  │  └─ C
  └─ D
     └─ E
  F
```

### 6.5 导入错误处理

| 错误类型 | 处理方式 |
|----------|----------|
| 层级跳跃 | 如层级从 1 直接跳到 3（缺少层级 2），报错提示 |
| 层级超过 5 | 报错，禁止导入第 6 级及以上任务 |
| 缺少层级字段 | 报错，层级为必填字段 |

### 6.6 导入前置任务处理

导入时不支持前置任务关系导入，原因：
- WBS 编码是用户视角的动态值，导入数据中的编码无法用于匹配
- 导出时不包含 UUID，无法建立关联

处理方式：导入后前置任务为空，需手动重新设置。

## 七、并发与一致性

### 7.1 并发策略

| 策略 | 说明 |
|------|------|
| 最后写入优先 | 不做冲突检测，后操作覆盖前操作 |
| 原因 | 简单实现，适合测试数据阶段 |

### 7.2 状态变更

| 场景 | 影响 |
|------|------|
| 任务状态变更（如审批通过） | 不影响编码，不触发重算 |

## 八、性能优化

### 8.1 缓存策略

| 策略 | 说明 |
|------|------|
| 缓存粒度 | 按用户缓存（全局视图，不区分项目） |
| 缓存键 | `wbs_code_cache:{user_id}:global` |
| 失效条件 | 任务增删改操作、权限变更事件 |
| 权限变更失效 | 用户角色变更、项目成员变更、组织架构调整时，主动失效相关用户缓存 |
| 准确性保证 | 宁可牺牲性能，不牺牲准确性 |

### 8.2 缓存流程

```
查询任务列表：
1. 检查缓存是否存在（缓存键：{user_id}:global）
2. 存在 → 直接返回缓存结果
3. 不存在 → 实时计算编码，缓存结果，返回

任务变更操作：
1. 执行数据库操作
2. 删除该用户的全局缓存
3. 下次查询时重新计算

权限变更事件：
1. 用户角色变更 → 删除该用户缓存
2. 项目成员变更 → 删除受影响用户缓存
3. 组织架构调整 → 删除受影响用户缓存
```

### 8.3 计算优化

由于 WBS 编码现在是全局计算，需要确保计算性能：

| 优化项 | 说明 |
|--------|------|
| 一次计算 | 所有可见任务一次性计算，避免按项目多次查询 |
| 内存缓存 | 使用 Map 结构快速查找父子关系 |
| 避免递归深度 | 最大 5 层，递归深度可控 |

## 九、编码计算算法

### 9.1 算法流程

```
用户查询任务列表时：
1. 获取用户可见范围内的所有任务（基于数据权限过滤）
2. 对所有任务统一计算 WBS 编码（不区分项目）
3. 按 parent_id 构建任务树
4. 每层按 sort_order（优先）或 created_at 排序
5. 深度遍历分配编码：
   - 根任务：1, 2, 3...（全局连续）
   - 子任务：父编码.序号（如 4.1, 4.2...）
6. 返回带编码的任务列表
```

### 9.2 伪代码

```typescript
function calculateWBSCodes(tasks: Task[]): Map<string, string> {
  const codeMap = new Map<string, string>();

  // 构建父子关系映射（不区分项目，统一计算）
  const childrenMap = new Map<string | null, Task[]>();
  tasks.forEach(task => {
    const parentId = task.parent_id;
    if (!childrenMap.has(parentId)) {
      childrenMap.set(parentId, []);
    }
    childrenMap.get(parentId)!.push(task);
  });

  // 每层排序
  childrenMap.forEach(children => {
    children.sort((a, b) => {
      if (a.sort_order !== null && b.sort_order !== null) {
        return a.sort_order - b.sort_order;
      }
      if (a.sort_order !== null) return -1;
      if (b.sort_order !== null) return 1;
      return new Date(a.created_at).getTime() - new Date(b.created_at).getTime();
    });
  });

  // 递归分配编码（全局连续，根任务从1开始）
  function assignCode(parentId: string | null, prefix: string): void {
    const children = childrenMap.get(parentId) || [];
    children.forEach((child, index) => {
      const code = prefix ? `${prefix}.${index + 1}` : `${index + 1}`;
      codeMap.set(child.id, code);
      assignCode(child.id, code);
    });
  }

  assignCode(null, '');
  return codeMap;
}
```

### 9.3 关键变更说明

| 变更项 | 旧方案 | 新方案 |
|--------|--------|--------|
| 计算范围 | 按项目分组计算 | 全局统一计算 |
| 根任务编号 | 每个项目从 1 开始 | 全局从 1 开始连续递增 |
| 缓存粒度 | 按用户+项目 | 按用户（全局） |

| 风险 | 对策 |
|------|------|

## 附录：决策记录

| 问题 | 决策 | 原因 |
|------|------|------|
| 编码格式 | 纯数字点分 `1.2.3` | 简洁直观 |
| 最大层级 | 5 级，禁止创建第 6 级 | 实现最简单 |
| 排序依据 | 手动优先 + 创建时间 | 灵活性与自动化平衡 |
| 编码触发 | 实时计算 | 准确性优先 |
| **全局编码规则** | **根任务全局连续递增，不区分项目** | **用户视图直观，符合认知习惯** |
| 移动行为 | 整体移动（含子任务） | 符合用户预期 |
| 移动时层级 | 自动重新计算 | 保持数据一致性 |
| 移动层级超限 | 禁止操作并提示 | 实现最简单 |
| 跨项目移动 | 允许但需确认 | 灵活可控 |
| 跨项目前置任务 | 提示用户是否保留 | 用户知情选择 |
| 删除有前置引用的任务 | 提示用户，确认后自动解除引用 | 用户知情，操作灵活 |
| 复制位置 | 插入选中行之后 | 用户明确意图 |
| 复制前置任务 | 提示用户是否保留 | 用户知情选择 |
| 并发策略 | 最后写入优先 | 简单实现 |
| 导出原则 | 所见即所导 | 简单直接 |
| 导入原则 | 导入即新建，层级定结构 | 不依赖编码，按层级还原树结构 |
| 导入父子关系 | 按层级顺序，用栈推断父任务 | 简单可靠 |
| 导入前置任务 | 不支持，需手动重新设置 | 编码是动态的，无法匹配 |
| 状态变更 | 不影响编码 | 简单直接 |
| 缓存粒度 | 按用户缓存（全局） | 配合全局编码规则 |
| 缓存失效 | 事件驱动，权限变更时主动失效 | 权限变更不频繁 |
