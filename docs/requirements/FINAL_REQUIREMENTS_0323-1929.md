## 1.2 业务概述

### 1.2.1 业务定位

**平台类型**: 企业级任务管理系统
**目标用户**: 100人团队
**部署方式**: 公司内网部署
**设备支持**: 仅桌面端

### 1.2.2 核心价值主张

1. **多项目并行管理**: 同时管理多个项目，支持快速切换
2. **时间驱动规划**: 可视化时间线、甘特图、里程碑管理
3. **数据驱动决策**: 丰富的数据可视化和分析报表
4. **实时协作**: 多人同时编辑，数据自动同步
5. **企业级安全**: 细粒度权限、审计日志、版本控制
6. **能力匹配**: 基于技能模型的智能任务分配

### 1.2.3 用户角色与权限

| 角色 | 中文名称 | 描述 | 可编辑WBS | 数据访问范围 |
|------|----------|------|-----------|------------|
| admin | 系统管理员 | 系统最高权限 | ✅ | 所有项目数据 |
| tech_manager | 技术经理 | 技术部门管理者 | ✅ | 所有技术项目数据 |
| dept_manager | 部门经理 | 业务部门管理者 | ✅ | 本部门项目数据 |
| engineer | 工程师 | 普通开发人员 | ✅* | 参与的项目数据 |

*注：工程师编辑计划相关字段（开始日期、结束日期、工期、前置任务、提前/落后天数）需要审批

### 2.1.3 用户场景

#### 场景1：创建新项目

**前置条件**:
- 您已登录系统
- 您具有创建项目权限

**操作步骤**:
1. 点击"新建项目"按钮
2. 填写项目基本信息（项目代号、项目名称、项目类型）
3. 设置项目时间范围（计划开始日期、计划结束日期）
4. 添加项目描述（可选）
5. 点击"创建"按钮

**预期结果**:
- 新项目创建成功
- 显示在项目列表中
- 您可以进入项目详情
- 系统记录操作日志

#### 场景2：编辑项目信息

**前置条件**:
- 项目已存在
- 您具有编辑权限

**操作步骤**:
1. 进入项目详情
2. 点击"编辑"按钮
3. 修改需要变更的信息
4. 点击"保存"按钮

**预期结果**:
- 项目信息更新成功
- 项目成员收到通知
- 系统记录变更历史
- 版本号自动+1

#### 场景3：管理项目成员

**前置条件**:
- 项目已存在
- 您是 admin/dept_manager/tech_manager

**操作步骤**:
1. 进入项目详情
2. 切换到"成员"标签
3. 点击"添加成员"按钮
4. 从组织架构选择要添加的成员
5. 点击"确认"按钮

**预期结果**:
- 成员添加到项目
- 成员可以访问项目数据
- 成员收到项目邀请通知

#### 规则1：项目成员权限

**规则说明**:
- 项目成员可以查看项目数据
- 项目成员不能编辑项目基本信息
- 只有 admin/dept_manager/tech_manager 可以编辑项目

**应用场景**:
- 普通成员查看项目进度和任务
- 技术经理修改项目时间范围、添加/移除成员
- 系统管理员调整项目状态

**示例**:
```
张三是项目成员 → 可以查看项目A的所有任务，但不能修改项目信息
李四是技术经理 → 可以修改项目A的时间范围、添加/移除成员
王五是系统管理员 → 可以删除项目A（如果项目A没有任务）
```

#### 规则2：项目数据隔离

**规则说明**:
- 您只能看到自己参与的项目的数据
- 项目之间数据完全隔离
- 系统管理员可以查看所有项目

**应用场景**:
- 保护敏感项目数据
- 确保数据安全
- 不同部门项目独立管理

**示例**:
```
张三参与项目A和B → 只能看到项目A和B的数据
李四只参与项目C → 只能看到项目C的数据
王五是系统管理员 → 可以看到所有项目（A、B、C、D...）
```

## 模块 #2: WBS任务管理

**优先级**: P0
**需求来源**: 📋 原始需求 + 🔄 共同增强
**实现状态**: ✅ 已完整实现

### 2.2.1 功能概述

支持WBS（工作分解结构）任务管理，包括24列复杂表格（24个数据列 + 操作列 = 25列）、任务依赖关系、计划变更审批等功能。这是系统的核心模块，用于分解和管理项目任务。

### 2.2.2 24列基本规格

| 列号 | 名称 | 编辑 | 说明 |
|------|------|------|------|
| 0 | 操作按钮 | - | 快捷操作列，有添加任务、编辑任务、删除任务、维护进展 |
| 1 | WBS等级 | 🔵可编 | 子任务自动+1级 |
| 2 | WBS编码 | 🟢只读 | 系统自动生成 |
| 3 | 任务描述 | ✍️可编 | 必填项 |
| 4 | 任务状态 | 🟡只读 | 系统自动判断 |
| 5 | Redmine链接 | 🔗超链接 | 仅根任务可填 |
| 6 | 负责人 | 👤下拉 | 关联组织架构 |
| 7 | 任务类型 | 🔴下拉 | 子任务继承父任务，关联任务类型 |
| 8 | 优先级 | 🔴下拉 | 紧急/高/中/低 |
| 9 | 前置任务 | 🔗WBS编号 | 输入WBS编码 |
| 10 | 提前/落后 | ✍️天数 | 可正可负 |
| 11 | 开始日期 | 🟢📅选择 | 无前置任务时可编辑 |
| 12 | 工期 | 🟢天数 | +单休勾选框，鼠标悬浮显示"勾选表示工作6天" |
| 13 | 结束日期 | 🟢计算 | 根据工期计算 |
| 14 | 计划周期 | 🟢计算 | 不可编辑 |
| 15 | 预警天数 | 🔵可编 | 用于计算"延期预警"状态，默认3天 |
| 16 | 实际开始 | 🔵📅可编 | 手动填写 |
| 17 | 实际结束 | 🔵📅可编 | 手动填写 |
| 18 | 实际工期 | 🔵计算 | 不可编辑 |
| 19 | 全职比 | 🟢0-100% | 默认100 |
| 20 | 实际周期 | 🟢计算 | 不可编辑 |
| 21 | 项目 | 下拉选择 | 必须选择项目，关键项目 |
| 22 | 延期次数 | 🔴数字 | 点击查看详情。延期后计划未刷新则不累加，刷新后再次超期才累加 |
| 23 | 计划调整 | 🔵数字 | 点击查看详情 |
| 24 | 进展记录 | 🟣数字 | 点击查看详情 |

### 2.2.11 业务规则

#### 规则1：WBS编码自动生成

**规则说明**:
- WBS编码由系统自动生成，您不可手动编辑
- 编码格式：`父级编码.当前序号`
- 一级任务编码格式：`1`、`2`、`3`...
- 二级任务编码格式：`1.1`、`1.2`、`1.3`...
- 三级任务编码格式：`1.1.1`、`1.1.2`、`1.1.3`...

**应用场景**:
- 创建新任务时自动生成
- 删除任务时不重新编号（保持历史）
- 移动任务时重新计算编码

**示例**:
```
任务1（编码：1）
  ├── 任务1.1（编码：1.1）
  │   └── 任务1.1.1（编码：1.1.1）
  └── 任务1.2（编码：1.2）
任务2（编码：2）
```

#### 规则2：开始日期与前置任务的关系

**规则说明**:
- 当任务有前置任务时，开始日期由系统自动计算
- 计算公式：`开始日期 = 前置任务结束日期 + 提前/落后天数`
- 有前置任务时，开始日期显示为计算后的日期且不可编辑
- 删除前置任务后，开始日期恢复可编辑

**应用场景**:
- 设置前置任务后自动调整开始日期
- 修改前置任务的结束日期时，后续任务自动调整
- 清除前置任务关系后可手动设置日期

**示例**:
```
前置任务：2024-01-01 结束
提前/落后：2天
当前任务开始日期自动计算为：2024-01-03
```

#### 规则3：结束日期计算

**规则说明**:
- 当工期 > 0 时，结束日期由系统自动计算
- 计算时需考虑节假日（从系统配置读取）
- 计算公式：`结束日期 = 开始日期 + 工期 - 1`
- 有工期时，结束日期根据开始日期自动计算完成日期
- 工期为0或空时，可手动设置结束日期

**应用场景**:
- 设置工期后自动计算结束日期
- 修改开始日期时，结束日期自动调整
- 勾选"单休日"时按每周6天计算
- 鼠标选中勾选框提示"勾选表示每周工作6天"

**示例**:
```
开始日期：2024-01-01（周一）
工期：5天（双休日）
结束日期：2024-01-05（周五）

开始日期：2024-01-01（周一）
工期：5天（单休日）
结束日期：2024-01-06（周六）
```

#### 规则4：任务状态自动判断

**判断逻辑**（依次判断1-9，第一个满足条件的就是最终状态）:

| 序号 | 状态 | 判断条件 |
|------|------|---------|
| 1 | 待审批 | 上级主管未处理 |
| 2 | 已驳回 | 上级主管审批未通过 |
| 3 | 未开始 | 没有完整计划日期或有计划日期但无实际开始日期 |
| 4 | 进行中 | 有计划日期、实际开始日期，无实际完成日期且未超期 |
| 5 | 提前完成 | 有实际完成日期且早于计划完成日期 |
| 6 | **按时完成**  | **有实际完成日期且等于计划完成日期** |
| 7 | 延期预警 | 无实际完成日期且当前距离计划完成日期≤预警天数 |
| 8 | 已延迟 | 无实际完成日期且当前已超过计划完成日期 |
| 9 | 超期完成 | 有实际完成日期且晚于计划完成日期 |

#### 规则5：计划周期计算

**规则说明**:
- 计划周期由系统自动计算，您不可编辑
- 计算公式：`计划周期 = 结束日期 - 开始日期 + 1`
- 单位：天（日历天数）
- 显示在"计划周期"列

**应用场景**:
- 修改开始或结束日期时自动更新
- 用于统计分析项目周期

#### 规则5.5：实际工期计算

**规则说明**:
- 实际工期由系统自动计算，您不可编辑
- 计算公式：统计实际开始日期到实际结束日期之间的**工作日数**（跳过周末和节假日）
- 单位：天（工作日）
- 只有当实际开始和实际结束都有值时才计算

**应用场景**:
- 任务完成后统计实际投入的工作时间
- 用于工作量分析和效率评估
- 与实际周期对比可分析非工作日影响

#### 规则6：实际周期计算

**规则说明**:
- 实际周期由系统自动计算，您不可编辑
- 计算公式：`实际周期 = 实际结束日期 - 实际开始日期 + 1`
- 单位：天（日历天数，含周末和节假日）
- 只有当实际开始和实际结束都有值时才计算

**与实际工期的区别**:
- 实际周期 = 日历天数（含周末节假日）
- 实际工期 = 工作日数（跳过周末节假日）

**应用场景**:
- 任务完成后统计实际耗时
- 用于对比计划与实际的差异

#### 规则7：全职比影响

**规则说明**:
- 全职比表示任务投入的人力资源比例
- 取值范围：0-100
- 100表示全职，50表示半职
- 用于资源统计和负荷分析

**应用场景**:
- 统计人员的工作负荷
- 分析资源分配是否合理

### 2.8.5 业务规则 - 延期管理

#### 规则1：延期自动判断与智能累计

**规则说明**:
- 系统每日凌晨1点自动检查所有任务
- 判断条件：当前日期 > 计划结束日期 且 任务状态不是"已完成"
- 自动标记为"已延迟"

**延期次数智能累计规则**:
- **首次延期**：任务首次超过计划结束日期时，延期次数 = 1
- **持续延期不累加**：计划未刷新期间，延期次数保持不变
- **刷新后再次超期**：用户刷新计划后，任务再次超过新的结束日期时，延期次数 +1
- **计数含义**：延期次数记录的是"延期了几轮"，而非"延期了多少天"

**判断逻辑（代码示例）**:
```javascript
// 延期判断逻辑（智能累计）
function checkTaskDelay(task) {
  const today = new Date();
  const plannedEndDate = new Date(task.planned_end_date);

  if (today > plannedEndDate && task.status !== 'completed') {
    // 标记为延期
    task.status = 'delayed';

    // 智能累计：仅当上次检查时任务未延期，或刷新计划后首次超期时才累加
    if (!task.was_delayed_since_last_refresh) {
      task.delay_count += 1;
      task.was_delayed_since_last_refresh = true;
    }

    // 发送延期通知
    sendDelayNotification(task);
  }
}

// 计划刷新时重置延期标记
function onPlanRefresh(task) {
  task.was_delayed_since_last_refresh = false;
  // ... 其他刷新逻辑
}
```

#### 规则2：延期原因记录

**规则说明**:
- 工程师和技术经理可以添加延期原因
- 延期原因只能增加，不能删除或修改
- 每条记录包含：延期时间、延期原因、延期天数、记录人

**示例**:

**示例**:

**功能概述**:

**设计原则**:

##### 17.1 核心决策汇总

| # | 决策点 | 最终选择 | 理由 |
|---|--------|----------|------|
| 1 | 配置入口位置 | 分离式 | 模型配置在设置中，成员评估在成员管理中 |
| 2 | 层级结构 | 两级结构 | 模型→维度，保持简单便于快速迭代 |
| 3 | 成员关联模型 | 可以多次 | 一个人可同时做嵌入式和质量管理等工作 |
| 4 | 维度字段设计 | 简化版 | 名称、权重(%)、0-100分 |
| 5 | 任务分配建议 | 手动触发 | 点击"智能推荐"按钮显示推荐列表 |
| 6 | 报表能力展示 | 分维度展示 | 列表展示各维度得分 |
| 7 | 雷达图交互 | 点击展开 | 点击成员节点后展开显示雷达图 |

##### 17.2 两级结构定义

```
能力模型（Model）              ← 如："嵌入式开发能力"
├── 模型名称
├── 模型描述（可选）
├── 评价维度（可自定义多个）
│   ├── 维度名称：如"固件开发"
│   ├── 维度权重：如35%
│   └── 评分范围：0-100
└── 创建时间/更新时间
```

**数据约束**:
1. 权重约束：同一模型下所有维度权重之和必须等于100%
2. 评分范围：每个维度得分范围为0-100
3. 成员关联：同一成员可以关联同一模型多次，通过 `association_label` 区分
4. 维度数量：每个模型至少1个维度，最多10个维度

##### 17.3 成员能力关联

**关联特点**:
- 一个成员可以关联同一个模型多次
- 通过 `association_label` 区分同一模型的不同工作方向
- 例如：张三可以同时有"嵌入式开发能力-固件方向"和"嵌入式开发能力-驱动方向"

**评定记录字段**:

| 字段名称 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| id | UUID | ✅ | 主键 |
| user_id | String | ✅ | 成员ID |
| model_id | String | ✅ | 关联的能力模型ID |
| model_name | String | ✅ | 冗余字段，便于查询 |
| association_label | String | ❌ | 关联说明（区分同一模型多次关联） |
| dimension_scores | JSON | ✅ | 各维度得分数组 |

##### 17.4 智能推荐机制

### 3.3.4 数据访问规则

### 规则1：项目数据隔离

- 项目成员关系由 admin/dept_manager/tech_manager 设置

```
张三（engineer）参与项目A和B → 只能看到项目A和B的数据
```

**规则说明**:
- 工程师修改计划 → 需要审批
