## 1. 组织架构管理

### 1.1 部门树结构

**来源**: FINAL_REQUIREMENTS#L914-921, L984-999

#### 功能需求
| 功能 | 描述 |
|------|------|
| 部门树管理 | 支持多层级部门树结构 |
| 添加部门 | 在指定父部门下创建子部门 |
| 编辑部门 | 修改部门名称、上级部门 |
| 删除部门 | 删除部门（需无成员） |
| 查看组织架构 | 可视化树形展示 |

#### 验收标准
- 部门树结构正确展示
- 支持无限层级嵌套
- 拖拽调整部门层级（可选）

#### 组织架构关系示例
```
部门
├── 部门经理（dept_manager）
│   ├── 技术组1
│   │   ├── 技术经理（tech_manager）
│   │   ├── 工程师A
│   │   └── 工程师B
│   ├── 技术组2
│   │   ├── 技术经理（tech_manager）
│   │   ├── 工程师C
│   │   └── 工程师D
│   └── ...
```

## 3. 能力模型管理

### 3.1 概述

**来源**: FINAL_REQUIREMENTS#L1796-1809

| 属性 | 说明 |
|------|------|
| 优先级 | P1 |
| 实现状态 | ⏳ 待开发 |
| 核心决策 | 两级结构（模型→维度） |

#### 设计原则
- **KISS**: 采用两级简化结构，避免过度复杂
- **YAGNI**: 仅实现当前需要的功能，不做过度设计
- **DRY**: 能力模型可复用，成员可关联多个模型

### 3.2 核心决策汇总

**来源**: FINAL_REQUIREMENTS#L1810-1821

| # | 决策点 | 最终选择 | 理由 |
|---|--------|----------|------|
| 1 | 配置入口位置 | 分离式 | 模型配置在设置中，成员评估在成员管理中 |
| 2 | 层级结构 | 两级结构 | 模型→维度，保持简单便于快速迭代 |
| 3 | 成员关联模型 | 可以多次 | 一个人可同时做嵌入式和质量管理等工作 |
| 4 | 维度字段设计 | 简化版 | 名称、权重(%)、0-100分 |
| 5 | 任务分配建议 | 手动触发 | 点击"智能推荐"按钮显示推荐列表 |
| 6 | 报表能力展示 | 分维度展示 | 列表展示各维度得分 |
| 7 | 雷达图交互 | 点击展开 | 点击成员节点后展开显示雷达图 |

### 3.3 两级结构定义

**来源**: FINAL_REQUIREMENTS#L1822-1840

```
能力模型（Model）              ← 如："嵌入式开发能力"
├── 模型名称
├── 模型描述（可选）
├── 评价维度（可自定义多个）
│   ├── 维度名称：如"固件开发"
│   ├── 维度权重：如35%
│   └── 评分范围：0-100
└── 创建时间/更新时间
```

#### 数据约束
| 约束项 | 规则 |
|--------|------|
| 权重约束 | 同一模型下所有维度权重之和必须等于100% |
| 评分范围 | 每个维度得分范围为0-100 |
| 成员关联 | 同一成员可以关联同一模型多次，通过association_label区分 |
| 维度数量 | 每个模型至少1个维度，最多10个维度 |

### 3.4 成员能力关联

**来源**: FINAL_REQUIREMENTS#L1841-1862

#### 关联特点
- 一个成员可以关联同一个模型多次
- 通过`association_label`区分同一模型的不同工作方向
- 例如：张三可以同时有"嵌入式开发能力-固件方向"和"嵌入式开发能力-驱动方向"

#### 评定记录字段

| 字段名称 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| id | UUID | ✅ | 主键 |
| user_id | String | ✅ | 成员ID |
| model_id | String | ✅ | 关联的能力模型ID |
| model_name | String | ✅ | 冗余字段，便于查询 |
| association_label | String | ❌ | 关联说明（区分同一模型多次关联） |
| dimension_scores | JSON | ✅ | 各维度得分数组 |
| overall_score | Integer | ✅ | 加权平均分（计算得出） |
| evaluated_at | Date | ✅ | 评定时间 |
| evaluated_by | String | ✅ | 评定人ID |
| notes | Text | ❌ | 备注 |

### 3.5 智能推荐机制

**来源**: FINAL_REQUIREMENTS#L1863-1881

#### 任务类型-能力模型映射

| 任务类型 | 关联模型 | 优先级 |
|----------|----------|--------|
| 固件 | 嵌入式开发能力 | 1 |
| 板卡 | 嵌入式开发能力 | 1 |
| 驱动 | 嵌入式开发能力 | 1 |
| 接口类 | 系统设计能力 | 1 |
| 系统设计 | 系统设计能力 | 1 |
| 职能任务 | 通用能力 | 1 |

#### 推荐逻辑
```
1. 根据任务类型查询映射表，获取关联的能力模型列表（按优先级排序）
2. 遍历模型列表，查找有该模型能力评定的成员
3. 如果成员有多个同一模型的评定，取综合分数最高的那个
4. 按综合分数降序排列返回推荐列表
```

**来源**: FINAL_REQUIREMENTS#L1882-1907

|------|------|

| 功能 | 描述 |
|------|------|

| 格式 | `模型名称: 维度1:分数 | 维度2:分数 | 维度3:分数` |

| 功能 | 描述 |

## 接口需求

### API列表

#### 组织架构
```
GET    /api/departments              # 获取部门树
GET    /api/departments/:id          # 获取部门详情
POST   /api/departments              # 创建部门
PUT    /api/departments/:id          # 更新部门
DELETE /api/departments/:id          # 删除部门
```

#### 成员管理
```
GET    /api/members                  # 获取成员列表
GET    /api/members/:id              # 获取成员详情
POST   /api/members                  # 创建成员
PUT    /api/members/:id              # 更新成员
DELETE /api/members/:id              # 删除成员
```

#### 能力模型
```
GET    /api/capability-models        # 获取能力模型列表
GET    /api/capability-models/:id    # 获取能力模型详情
POST   /api/capability-models        # 创建能力模型
PUT    /api/capability-models/:id    # 更新能力模型
DELETE /api/capability-models/:id    # 删除能力模型
```

#### 成员能力评定
```
GET    /api/members/:id/capabilities           # 获取成员能力列表
POST   /api/members/:id/capabilities           # 添加成员能力评定
PUT    /api/members/:id/capabilities/:capId    # 更新成员能力评定
DELETE /api/members/:id/capabilities/:capId    # 删除成员能力评定
```

#### 智能推荐
```
GET    /api/tasks/recommend-assignee   # 获取任务负责人推荐列表
```

#### 任务类型-能力模型映射
```
GET    /api/config/task-type-mappings  # 获取任务类型与能力模型的映射关系
POST   /api/config/task-type-mappings  # 创建任务类型-能力模型映射
```
