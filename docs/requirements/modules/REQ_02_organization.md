## 1. 组织架构管理

### 1.1 部门树结构

**来源**: FINAL_REQUIREMENTS_0404-0827#L914-921, L984-999

#### 功能需求
| 功能 | 描述 |
|------|------|
| 部门树管理 | 支持多层级部门树结构 |
| 添加部门 | 在指定父部门下创建子部门 |
| 编辑部门 | 修改部门名称、上级部门 |
| 删除部门 | 删除部门（需无成员） |
| 查看组织架构 | 可视化树形展示 |

#### 验收标准
- 部门树结构正确展示
- 支持无限层级嵌套
- 拖拽调整部门层级（可选）

#### 组织架构关系示例

> **核心逻辑**：
> - 每个部门都有部门经理（dept_manager）
> - 部门下有成员和子部门（同级关系）
> - 子部门的负责人也是部门经理
> - 技术组的负责人是技术经理（tech_manager）

```
📁 技术中心 [35]                    ← 顶级部门，35人（含所有下级）
   👤 张三 [部门经理] 👑             ← 部门负责人，👑金色皇冠
   📁 研发一部 [12]                 ← 子部门，12人
      👤 李四 [部门经理] 👑          ← 子部门负责人
      📁 技术组A [5]                ← 技术组，5人
         👤 王五 [技术经理] ⭐       ← 技术组负责人，⭐蓝色星形
         👤 赵六 [工程师]
         👤 钱七 [工程师]
      📁 技术组B [7]                ← 技术组，7人
         👤 孙八 [技术经理] ⭐
         ...
   📁 研发二部 [15]                 ← 子部门，15人
      👤 周九 [部门经理] 👑
      ...
```

#### 管理角色徽标规范

| 角色 | 徽标 | 颜色 | 说明 |
|------|------|------|------|
| 系统管理员 | 🛡️ Shield | 红色 | 系统权限管理者 |
| 部门经理 | 👑 Crown | 金色/琥珀色 | 部门/子部门负责人 |
| 技术经理 | ⭐ Star | 蓝色/紫色 | 技术组负责人 |

3. 填写成员信息（姓名、工号、岗位）

- 成员添加到部门
- 自动创建登录账号
- 显示初始密码（可复制）
- 成员可以使用工号和初始密码登录

### 2.5 成员删除管理

> **新增时间**: 2026-03-23
> **优先级**: P1

#### 删除类型

| 类型 | 说明 | 权限 |
|------|------|------|
| **软删除（停用）** | 设置 is_active=false，保留历史数据 | admin, dept_manager |
| **物理删除** | 从数据库彻底删除记录 | admin only |

#### 删除前检查项

删除用户前需显示以下关联数据统计：

| 检查项 | 说明 | 物理删除约束 |
|--------|------|--------------|
| 是否为部门经理 | 如果是部门经理，需先移除或转移 | ⚠️ 阻止删除 |
| 参与的项目数 | 作为成员参与的项目数量 | ⚠️ 需先退出项目 |
| 负责任务数 | 作为负责人/执行人的任务数量 | ⚠️ 需先移交任务 |
| 审批记录数 | 发起的审批记录数量 | ✅ 允许删除（保留记录） |
| 能力评估记录 | 关联的能力评估数据 | ✅ 允许删除（级联删除） |

#### 删除确认流程

```
1. 用户点击"删除"按钮
2. 系统检查用户关联数据
3. 显示关联数据统计面板：
   ┌─────────────────────────────────┐
   │  删除确认                        │
   ├─────────────────────────────────┤
   │  用户：张三 (10001)              │
   │                                 │
   │  关联数据统计：                  │
   │  • 参与项目：3 个                │
   │  • 负责任务：5 个                │
   │  • 审批记录：12 条               │
   │                                 │
   │  ⚠️ 该用户有进行中的任务，        │
   │     请先移交任务后再删除         │
   ├─────────────────────────────────┤
   │  [取消]  [确认停用]  [物理删除]   │
   └─────────────────────────────────┘
4. 用户选择删除类型并确认
5. 执行删除操作
```

#### API 接口

```bash
# 软删除（停用用户）
PUT /api/members/:id/deactivate

# 物理删除（仅 admin）
DELETE /api/members/:id

# 删除前检查（获取关联数据统计）
GET /api/members/:id/deletion-check
```

#### 响应示例（删除检查）

```json
{
  "canDelete": false,
  "canDeactivate": true,
  "warnings": [
    "该用户有 5 个进行中的任务"
  ],
  "blockingReasons": [
    "用户是「研发一部」的部门经理"
  ],
  "stats": {
    "projects": 3,
    "tasks": 5,
    "approvals": 12,
    "capabilityRecords": 2
  }
}
```

#### 数据库字段扩展

| 字段名称 | 类型 | 说明 |
|---------|------|------|
| deleted_at | Timestamp | 软删除时间（null 表示未删除） |

### 2.6 用户与组织架构关联规则

> **新增时间**: 2026-03-23
> **优先级**: P0（核心业务规则）

#### 核心规则

| 用户类型 | 创建方式 | 组织架构要求 |
|----------|----------|--------------|
| **内置用户** | 系统初始化时创建 | 无需关联组织架构 |
| **普通用户** | 通过组织架构创建 | 必须先在组织架构中存在 |

#### 内置用户定义

| 用户名 | 角色 | 用途 |
|--------|------|------|
| admin | admin | 系统管理员，用于系统初始化和紧急维护 |

#### 内置用户特征
- 由数据库迁移脚本在系统初始化时创建
- 不需要关联部门（`department_id` 可为 null）
- 不可通过常规成员管理流程删除
- 首次登录后应强制修改密码

#### 普通用户创建流程

```
1. 在组织架构中选择部门
2. 添加成员（填写姓名、工号、角色）
3. 系统自动创建登录账号
4. 生成初始密码
5. 成员可使用工号+初始密码登录
```

#### 数据库约束

| 字段 | 内置用户 | 普通用户 |
|------|----------|----------|
| department_id | 允许 NULL | 必须有值 |
| is_builtin | true | false |
| created_by | 'system' | 操作人ID |

#### 新增字段

| 字段名称 | 类型 | 说明 |
|---------|------|------|
| is_builtin | Boolean | 是否为内置用户（默认 false） |

#### API 验证规则

- 创建成员时，如果 `department_id` 为空且非内置用户创建接口，返回错误
- 删除内置用户时，返回"内置用户不可删除"错误

#### 内置用户可见性规则

| 角色 | 可见性 | 说明 |
|------|--------|------|
| admin | ✅ 可见 | 系统管理员可以看到所有内置用户 |
| dept_manager | ❌ 不可见 | 部门经理看不到内置用户 |
| tech_manager | ❌ 不可见 | 技术经理看不到内置用户 |
| engineer | ❌ 不可见 | 工程师看不到内置用户 |

**实现要求：**
- 组织架构 API 返回成员列表时，非 admin 角色自动过滤 is_builtin=true 的用户
- 成员选择器组件中，非 admin 角色不显示内置用户

## 3. 能力模型管理

### 3.1 概述

**来源**: FINAL_REQUIREMENTS_0404-0827#L1796-1809

| 属性 | 说明 |
|------|------|
| 优先级 | P1 |
| 实现状态 | ⏳ 待开发 |
| 核心决策 | 两级结构（模型→维度） |

#### 设计原则
- **KISS**: 采用两级简化结构，避免过度复杂
- **YAGNI**: 仅实现当前需要的功能，不做过度设计
- **DRY**: 能力模型可复用，成员可关联多个模型

### 3.2 核心决策汇总

**来源**: FINAL_REQUIREMENTS_0404-0827#L1810-1821

| # | 决策点 | 最终选择 | 理由 |
|---|--------|----------|------|
| 1 | 配置入口位置 | 分离式 | 模型配置在设置中，成员评估在成员管理中 |
| 2 | 层级结构 | 两级结构 | 模型→维度，保持简单便于快速迭代 |
| 3 | 成员关联模型 | 可以多次 | 一个人可同时做嵌入式和质量管理等工作 |
| 4 | 维度字段设计 | 简化版 | 名称、权重(%)、0-100分 |
| 5 | 任务分配建议 | 手动触发 | 点击"智能推荐"按钮显示推荐列表 |
| 6 | 报表能力展示 | 分维度展示 | 列表展示各维度得分 |
| 7 | 雷达图交互 | 点击展开 | 点击成员节点后展开显示雷达图 |

### 3.3 两级结构定义

**来源**: FINAL_REQUIREMENTS_0404-0827#L1822-1840

```
能力模型（Model）              ← 如："嵌入式开发能力"
├── 模型名称
├── 模型描述（可选）
├── 评价维度（可自定义多个）
│   ├── 维度名称：如"固件开发"
│   ├── 维度权重：如35%
│   └── 评分范围：0-100
└── 创建时间/更新时间
```

#### 数据约束
| 约束项 | 规则 |
|--------|------|
| 权重约束 | 同一模型下所有维度权重之和必须等于100% |
| 评分范围 | 每个维度得分范围为0-100 |
| 成员关联 | 同一成员可以关联同一模型多次，通过association_label区分 |
| 维度数量 | 每个模型至少1个维度，最多10个维度 |

### 3.4 成员能力关联

**来源**: FINAL_REQUIREMENTS_0404-0827#L1841-1862

#### 关联特点
- 一个成员可以关联同一个模型多次
- 通过`association_label`区分同一模型的不同工作方向
- 例如：张三可以同时有"嵌入式开发能力-固件方向"和"嵌入式开发能力-驱动方向"

#### 评定记录字段

| 字段名称 | 类型 | 必填 | 说明 |
|---------|------|------|------|
| id | UUID | ✅ | 主键 |
| user_id | String | ✅ | 成员ID |
| model_id | String | ✅ | 关联的能力模型ID |
| model_name | String | ✅ | 冗余字段，便于查询 |
| association_label | String | ❌ | 关联说明（区分同一模型多次关联） |
| dimension_scores | JSON | ✅ | 各维度得分数组 |
| overall_score | Integer | ✅ | 加权平均分（计算得出） |
| evaluated_at | Date | ✅ | 评定时间 |
| evaluated_by | String | ✅ | 评定人ID |
| notes | Text | ❌ | 备注 |

### 3.5 智能推荐机制

**来源**: FINAL_REQUIREMENTS_0404-0827#L1863-1881

#### 任务类型-能力模型映射

| 任务类型 | 关联模型 | 优先级 |
|----------|----------|--------|
| 固件 | 嵌入式开发能力 | 1 |
| 板卡 | 嵌入式开发能力 | 1 |
| 驱动 | 嵌入式开发能力 | 1 |
| 接口类 | 系统设计能力 | 1 |
| 系统设计 | 系统设计能力 | 1 |
| 职能任务 | 通用能力 | 1 |

#### 推荐逻辑
```
1. 根据任务类型查询映射表，获取关联的能力模型列表（按优先级排序）
2. 遍历模型列表，查找有该模型能力评定的成员
3. 如果成员有多个同一模型的评定，取综合分数最高的那个
4. 按综合分数降序排列返回推荐列表
```
