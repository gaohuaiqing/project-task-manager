## 🔴 严重风险问题（立即处理）

### 1. Redis 服务强依赖问题

**位置**: `app/server/src/index.ts:2124`

**问题描述**:
- 服务启动时**强制要求** Redis 连接成功
- 如果 Redis 不可用，整个服务无法启动
- 虽然有 `REDIS_OPTIONAL` 环境变量，但默认未启用

**代码证据**:
```typescript
// 初始化 Redis 缓存（必须，失败时终止启动）
await redisCacheService.init();
console.log('[服务器] ✅ Redis 缓存初始化成功');
```

**影响**:
- Redis 服务停止 → 后端服务完全离线
- Redis 网络抖动 → 服务重启失败
- 单点故障风险极高

**解决方案**:
1. 启用 Redis 可选模式：设置 `REDIS_OPTIONAL=true`
2. 实现 Redis 降级机制：Redis 不可用时使用内存缓存
3. 添加 Redis 自动重连逻辑

### 2. 数据库连接池配置不当

**位置**: `app/server/src/services/DatabaseService.ts:19-24`

**问题描述**:
- 连接池大小过小：生产环境仅50个连接
- 获取连接超时过短：8000ms
- 队列限制不够：100个请求

**配置值分析**:
```typescript
connectionLimit: process.env.NODE_ENV === 'production' ? 50 : 30
queueLimit: 100
acquireTimeout: 8000  // 8秒获取连接超时
```

**影响场景**:
- 高并发时连接池耗尽 → 请求被拒绝
- 慢查询占用连接 → 新请求超时
- 队列积压 → 响应时间急剧增加

**实际影响计算**:
```
假设：每个查询平均 500ms
最大并发处理能力 = 50 连接 × (1000ms / 500ms) = 100 QPS

实际需求：项目管理模块通常需要 200-500 QPS
缺口：100-400 QPS
```

**解决方案**:
1. 增加连接池大小：生产环境改为 100-200
2. 增加队列限制：200-500
3. 优化慢查询，减少查询时间
4. 实现连接池监控和自动扩展

---

### 3. 查询超时设置过于激进

**位置**: `app/server/src/utils/DatabaseQueryTimeout.ts:22-28`

**问题描述**:
- 默认查询超时仅 3000ms（3秒）
- 短查询超时仅 1000ms（1秒）
- 复杂业务查询可能超时

**超时配置**:
```typescript
export const QUERY_TIMEOUT = {
  DEFAULT: 3000,      // 默认超时：3秒
  SHORT: 1000,        // 短查询：1秒
  MEDIUM: 5000,       // 中等查询：5秒
  LONG: 10000,        // 长查询：10秒
  BATCH: 30000,       // 批量查询：30秒
  ANALYTICS: 60000    // 分析查询：60秒
};
```

**影响**:
- 项目列表查询可能涉及多表 JOIN → 超时
- WBS 任务树形查询可能超过 3 秒 → 超时
- 统计分析查询经常失败 → 数据不准确

**实际案例分析**:
```sql
-- 典型项目查询（可能超过3秒）
SELECT t.*, p.name, m.name, parent.task_name
FROM wbs_tasks t
LEFT JOIN projects p ON t.project_id = p.id
LEFT JOIN members m ON t.assignee_id = m.id
LEFT JOIN wbs_tasks parent ON t.parent_id = parent.id
WHERE t.project_id = ? AND t.status != 'deleted'
ORDER BY t.created_at DESC
LIMIT 100
```

**解决方案**:
1. 调整默认超时为 5000-8000ms
2. 为不同操作类型设置合适的超时
3. 优化 SQL 查询，添加必要索引
4. 实现查询结果缓存

### 4. 前端请求超时不合理

**位置**: `app/src/services/ApiService.ts:23-24`

**问题描述**:
- 前端请求超时设置为 10000ms（10秒）
- 但后端处理时间可能超过 10 秒
- 导致前端显示"服务离线"

**代码证据**:
```typescript
// 优化：请求超时控制，默认10秒
private readonly REQUEST_TIMEOUT = 10000;
```

**实际影响**:
```
用户体验时间线：
0s   - 用户发起请求
3s   - 后端查询开始
5s   - 数据处理中
8s   - 准备响应
10s  - ❌ 前端超时，显示"离线"
12s  - 后端实际完成（但前端已放弃）
```

**解决方案**:
1. 前端超时调整为 30000ms（30秒）
2. 实现请求进度提示
3. 添加请求重试机制
4. 区分不同操作的超时时间

### 5. WebSocket 连接数限制

**位置**: `app/server/src/index.ts:216`

**问题描述**:
- 最大 WebSocket 连接数限制为 1000
- 每个用户可能打开多个标签页
- 连接数达到上限后新用户无法连接

**配置**:
```typescript
const MAX_WS_CONNECTIONS = 1000;
```

**实际影响计算**:
```
假设：
- 每个用户平均 3 个标签页（项目列表、详情、任务管理）
- 实际支持用户数 = 1000 / 3 = 333 用户

超过 333 用户后，新连接被拒绝 → 显示"服务离线"
```

**解决方案**:
1. 增加连接数限制到 5000-10000
2. 实现连接池管理
3. 优化单连接复用
4. 实现连接自动清理
