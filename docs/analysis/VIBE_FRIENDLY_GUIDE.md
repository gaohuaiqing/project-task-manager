# 如何编写 AI 友好的设计文档

> **原则**: 描述体验和意图，让 AI 推断实现细节
> **避免**: 技术术语、具体数值、实现细节

---

## ✅ 好的描述方式

### 描述用户感受

❌ **不好的写法**：
```
对话框使用 max-w-3xl max-h-[95vh]
```

✅ **好的写法**：
```
对话框应该宽敞舒适，占据屏幕大部分空间，
让用户能够一眼看到表单的整体结构
```

### 描述相对关系

❌ **不好的写法**：
```
日期字段使用 grid-cols-2 布局
```

✅ **好的写法**：
```
日期字段应该并排显示，左右对称，
这样用户可以快速对比开始和结束时间
```

### 描述行为

❌ **不好的写法**：
```
DialogHeader 使用 sticky top-0
```

✅ **好的写法**：
```
对话框标题应该始终可见，
即使用户滚动表单内容，标题也不会消失
```

## 🎯 突出核心要求

使用引用块来强调不可妥协的要求：

```markdown
> ⚠️ **核心要求**: 对话框绝不能出现水平滚动条。
> 水平滚动会让用户感到困惑，不知道还有更多内容。
>
> 如果内容太多，只应该出现垂直滚动条。
```

使用"必须"和"禁止"来明确要求：

```markdown
**必须**:
- 对话框应该一屏显示大部分内容
- 时间计划编辑器必须是独立的对话框

**禁止**:
- 对话框不能出现水平滚动条
- 不能把时间计划编辑器嵌套在其他对话框里
```

## 📐 使用示意图的正确方式

### 确保示意图与文字一致

如果文档说"下拉框"，示意图就应该画下拉框：

```
┌──────────────────────────────────────┐
│ 项目类型: [下拉选择 ▼]              │ ← 清晰的下拉框
│           └─ 产品开发类              │
│           └─ 职能管理类              │
│           └─ 物料改代类              │
└──────────────────────────────────────┘
```

### 在示意图中标注重要要求

```
┌──────────────────────────────────────┐
│ 创建项目                        [×] │
├──────────────────────────────────────┤
│                                      │
│ ⚠️ 注意：此对话框应该一屏显示完整     │
│ 不要让用户频繁滚动                   │
│                                      │
│ [表单内容...]                        │
└──────────────────────────────────────┘
```

## 🔍 提供验收标准（从用户角度）

不要列出技术检查项，而是列出用户体验检查：

```markdown
## ✅ 验收标准

**打开对话框后，检查：**
1. 对话框看起来宽敞吗？还是拥挤？
2. 用户能一眼看到所有重要字段吗？
3. 有没有水平滚动条？（如果有，就是错的）
4. 标题在滚动时会消失吗？（如果会，就是错的）

**点击配置时间计划后，检查：**
1. 用户感知到这是新的对话框吗？
2. 新对话框可以独立关闭吗？
3. 关闭新对话框后，父对话框还在吗？
```

## 💬 与 AI 交流的技巧

### 技巧 1: 使用"用户应该"句式

这会引导 AI 从用户体验角度思考：

```
✅ "用户应该能够快速扫描所有表单字段"
❌ "表单应该使用 space-y-4 间距"
```

### 技巧 2: 描述问题而非解决方案

让 AI 推断最好的解决方案：

```
✅ "当前对话框太小，用户需要频繁滚动，这很烦人"
❌ "对话框高度应该是 95vh"
```

### 技巧 3: 描述相对关系

```
✅ "日期字段应该并排显示，方便用户对比"
❌ "使用 grid grid-cols-2"
```

### 技巧 4: 使用比喻

```
✅ "对话框应该像一张A4纸，内容一目了然"
❌ "对话框宽度应该是 768px"
```

## 🚫 避免的陷阱

### 陷阱 1: 混合技术术语和用户描述

❌ 不好的写法：
```
对话框使用 Dialog 组件，max-w-3xl，让用户感觉宽敞
```

✅ 好的写法：
```
对话框应该宽敞舒适，占据屏幕大部分空间
```

### 陷阱 2: 示意图与文字矛盾

❌ 不好的写法：
```
示意图：[卡片] [卡片]  ← 图示是卡片
文字：使用下拉框选择    ← 文字说下拉框
```

✅ 好的写法：
```
示意图：[下拉选择 ▼]   ← 图示是下拉框
文字：从下拉框选择     ← 文字说下拉框
```

### 陷阱 3: 把实现细节混入设计文档

❌ 不好的写法：
```
日期字段：使用 grid-cols-2，响应式布局，md:breakpoint
```

✅ 好的写法：
```
日期字段：左右并排显示，方便用户快速对比
```

## 📋 文档结构建议

```markdown
# 功能设计文档

## 🎯 核心原则（不可妥协）
> 用引用块列出最重要的用户体验要求

## 📐 设计示意
> 用 ASCII 图示展示布局
> 确保图示与文字描述一致

## 🧨 详细说明
> 描述每个部分的用户体验
> 使用"用户应该"句式

## ✅ 验收标准
> 从用户角度列出检查项
> 不要涉及技术细节

## 💬 修改指南
> 告诉用户如何修改文档
> 用自然语言描述
```

---

## 🎯 总结

**好的设计文档**：
- ✅ 描述用户体验和意图
- ✅ 使用自然语言，避免技术术语
- ✅ 突出核心要求（引用块、加粗）
- ✅ 从用户角度写验收标准
- ✅ 示意图与文字描述一致

**不好的设计文档**：
- ❌ 包含具体的技术实现细节
- ❌ 混合技术术语和用户描述
- ❌ 示意图与文字矛盾
- ❌ 关键要求被埋没在普通文字中
- ❌ 要求用户了解技术细节

记住：**您是产品经理，不是程序员**。描述您想要的体验，让 AI 处理实现细节。
