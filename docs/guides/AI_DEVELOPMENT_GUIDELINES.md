# AI 开发最佳实践指南

> **目的**: 减少 AI 幻觉导致的开发偏差，确保实现与需求文档一致
> **创建日期**: 2026-04-08
> **版本**: v1.0

---

## 一、问题背景

### 1.1 典型 AI 幻觉表现

| 表现 | 示例 |
|------|------|
| 声称完成但实际缺失 | 注释写"2x2网格"，代码只有1x2 |
| 框架完整但内容为空 | 组件存在，但数据全是空数组 |
| 部分实现当作全部 | 实现2个图表，需求要求4个 |
| 样式声称符合但偏差 | 需求12px圆角，实现8px |

### 1.2 根本原因

1. **缺乏验证机制** - 没有对照需求逐项检查
2. **任务粒度过大** - "实现仪表板"作为单一任务
3. **无完成标准** - 没有明确定义"什么叫完成"
4. **连续开发无暂停** - 多个功能连续开发，没有验证点

3. **全部勾选才能结束** - 不允许部分完成

## 三、分阶段开发流程

### 3.1 阶段划分原则

```
一个大功能 → 拆分为多个阶段 → 每阶段有明确验证点
```

### 3.2 阶段模板

```
## 阶段N: [阶段名称]

### 输入
- 需求文档: [路径]
- 检查清单: [路径或内嵌]

### 任务
1. [具体任务1]
2. [具体任务2]
3. [具体任务3]

### 验证点
- [ ] 对照检查清单逐项验证
- [ ] 所有组件已渲染（非空数据）
- [ ] 样式符合规范
- [ ] 布局符合需求

### 退出条件
- 检查清单100%勾选
- 验证点全部通过
```

## 四、强制验证机制

### 4.1 开发完成验证

**使用 `verification-before-completion` skill**：

```
我已完成 [功能名称] 的开发，请使用 verification-before-completion 验证：
1. 对照需求文档 [路径]
2. 检查所有组件是否实现
3. 检查数据是否填充（非空数组）
4. 检查样式是否符合规范
```

### 4.2 代码审查验证

**使用 `code-reviewer` agent**：

```
请使用 code-reviewer agent 审查 [功能名称]：
- 需求文档: [路径]
- 检查清单: [路径]
- 重点关注: 组件完整性、样式规范、布局结构
```

### 4.3 里程碑验证

**每个里程碑必须执行**：

```bash
# 1. 运行应用
npm run dev

# 2. 手动验证
- 打开对应页面
- 检查所有组件是否显示
- 检查数据是否正确

# 3. 自动化验证（如有测试）
npm run test -- --grep "[功能名称]"
```

## 五、需求文档最佳实践

### 5.1 文档粒度原则

**当前文档结构已经合理**：
```
docs/requirements/
├── FINAL_REQUIREMENTS.md        # 总体需求
├── UI_Requirement.md            # UI规格
└── modules/
    ├── REQ_07_analytics.md      # 模块主文档
    ├── REQ_07a_dashboard.md     # 子模块文档 ← 合适的粒度
    ├── REQ_07b_reports.md
    └── REQ_07c_system.md
```

**不建议继续拆分**：
- 子模块文档（如 REQ_07a_dashboard.md）已经是合适的粒度
- 再拆分会导致需求分散，AI需要读取多个文件
- 破坏需求的完整性和连贯性

### 5.2 在现有文档中强化可验证性

**方法：在需求文档中增加检查清单章节**

```markdown
# 模块需求：仪表板 (07a-dashboard)

## 现有需求内容...

---

## 实现检查清单 (AI 开发必读)

### 组件清单
| 组件 | 类型 | 需求行号 | 状态 |
|------|------|---------|------|
| 预警卡片x4 | 卡片 | L180-186 | ⬜ |
| 核心指标卡片x8 | 卡片 | L190-197 | ⬜ |
| ... | ... | ... | ... |

### 样式规范检查
| 元素 | 需求值 | 需求行号 | 状态 |
|------|--------|---------|------|
| 卡片圆角 | 12px | L59 | ⬜ |
| ... | ... | ... | ... |
```

### 5.3 检查清单独立文件（可选）

如果需求文档较长，可将检查清单独立：

```
docs/requirements/modules/
├── REQ_07a_dashboard.md      # 需求文档（不变）
└── REQ_07a_CHECKLIST.md      # 实现检查清单（新增）
```

**检查清单与需求文档的关系**：
- 检查清单是需求文档的"可执行版本"
- 每项都引用需求文档的具体行号
- 开发前创建，开发中勾选，开发后验证

## 六、TDD（测试驱动开发）

### 6.1 测试用例模板

```typescript
describe('[功能名称]', () => {
  describe('组件完整性', () => {
    it('应显示 [组件A]', () => {
      // 渲染组件
      // 检查组件是否存在
    });

    it('应显示 [组件B]', () => {
      // ...
    });
  });

  describe('样式规范', () => {
    it('[元素] 应为 [值]', () => {
      // 检查样式
    });
  });

  describe('数据完整性', () => {
    it('[字段] 不应为空', () => {
      // 检查数据
    });
  });
});
```

### 6.2 开发流程

```
1. 编写测试用例（基于需求）
2. 运行测试 → 失败（红）
3. 实现代码
4. 运行测试 → 通过（绿）
5. 重构优化
6. 重复步骤1-5
```

## 八、附录：常见AI幻觉模式

| 幻觉模式 | 表现 | 检查方法 |
|---------|------|---------|
| 注释与实现不符 | 注释说"2x2"，代码只有1x2 | 对照注释检查代码 |
| 组件存在但无数据 | `alerts: []` | 检查数据是否填充 |
| 部分实现 | 只实现2个图表，需求4个 | 对照需求文档计数 |
| 样式偏差 | 需求12px，实现8px | 对照样式规范检查 |
| 类型定义与实际不符 | 类型有字段，数据没有 | 检查类型与数据一致性 |
