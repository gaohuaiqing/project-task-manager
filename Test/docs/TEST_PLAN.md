## 一、测试目标

### 核心目标

**确保上线后用户可见的功能正常运作**

### 子目标

1. **功能正确性**：用户能完成核心业务操作（登录、创建项目、管理任务等）
2. **数据一致性**：操作后数据正确保存和更新
3. **安全性**：权限控制正确，无越权访问
4. **用户体验**：页面正常显示，交互流畅
5. **多角色协作**：不同角色用户可以同时使用系统，数据隔离正确

---

## 二、用户角色定义

| 角色 | 主要职责 | 典型使用场景 |
|------|---------|-------------|
| **admin** | 系统管理员 | 管理用户、配置系统、查看所有数据 |
| **dept_manager** | 部门经理 | 管理本部门项目、审批任务变更、查看部门报表、授权技术经理管理其他组 |
| **tech_manager** | 技术经理 | 管理本技术组任务、审批工程师提交的变更、管理被授权的其他技术组 |
| **engineer** | 工程师 | 执行任务、更新进度、提交变更申请 |

### 关键业务规则

- 部门经理可以将技术经理 A 授权管理技术经理 B 的组
- 技术经理被授权后，可以查看和管理被授权组的任务

## 三、核心业务流程

### 流程 1：用户登录与权限验证
```
用户登录 → 验证身份 → 加载对应角色权限 → 显示对应菜单和仪表板
```

### 流程 2：项目管理
```
创建项目 → 添加项目成员 → 设置里程碑 → 管理时间线
```

### 流程 3：任务管理（核心）
```
创建任务 → 编辑任务 → 删除任务 → 查看任务详情
```
**关键约束**：任务必须关联项目

### 流程 4：任务导入导出
```
导出任务模板 → 填写任务数据 → 导入任务 → 验证导入结果
```

### 流程 5：任务审批流程
```
工程师编辑任务计划 → 提交审批 → 技术经理/部门经理审批 → 生效/驳回
```

### 流程 6：多角色协作
```
技术经理 A 管理组 1 → 部门经理授权 → 技术经理 A 管理组 1 + 组 2
```

### 流程 7：报表查看与导出
```
选择报表类型 → 筛选条件 → 查看数据 → 导出 Excel
```

### 关键设计思想

- **项目管理与任务管理弱耦合**
- **任务管理是系统核心**
- **任务 CRUD 是核心中的核心**
- **联动自动计划调整**：设置前置任务 → 自动计算开始日期 → 级联更新后续任务

## 五、任务管理日常常用场景

### 高频操作（每天使用）

| 场景 | 说明 |
|------|------|
| **任务内容属性更新** | 修改任务描述、负责人、优先级等基本信息 |
| **维护任务进度** | 记录实际开始/结束时间、更新进展记录 |
| **查看任务列表** | 按项目/负责人/状态筛选、搜索任务 |
| **查看任务详情** | 点击任务查看完整信息、历史记录 |

### 中频操作（每周使用）

| 场景 | 说明 |
|------|------|
| **创建任务** | 创建根任务、创建子任务 |
| **调整时间计划** | 修改开始日期、工期、前置任务 |
| **联动自动计划调整** | 设置前置任务后，自动计算后续任务的开始日期 |
| **处理审批** | 提交变更申请、审批/驳回变更 |
| **处理延期任务** | 查看延期预警、更新延期任务状态 |

### 低频操作（偶尔使用）

| 场景 | 说明 |
|------|------|
| **删除任务** | 删除单个任务、批量删除 |
| **导入导出任务** | 批量导入、导出 Excel |
| **调整 WBS 结构** | 移动任务层级、调整顺序 |

## 八、测试执行方式

### 方案选择

**AI 实时操作浏览器（Chrome DevTools MCP）**

### 选择理由

1. **避免 Playwright 痛点**：无需维护选择器、无需搭建基础设施、无需编写脚本
2. **灵活性强**：能处理意外情况，能发现非预期问题
3. **探索性好**：AI 可以"感觉不对劲"并深入调查
4. **快速开始**：确认环境后立即执行

### 执行策略

- **分批执行**：按优先级分批执行，每批次完成后等待用户确认
- **状态追踪**：实时记录执行进度到 EXECUTION_STATE.md
- **Bug 管理**：发现问题记录到 BUG_LIST.md
- **截图存证**：每个操作截图保存
