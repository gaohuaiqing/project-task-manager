## 一、测试理念

### 1.1 核心原则

| 原则 | 说明 |
|------|------|
| **AI驱动** | AI作为测试执行者，实时探索+验证，而非机械按文档执行 |
| **精简预设** | 仅预设~123个核心路径用例，边界和异常由AI实时生成 |
| **结构化指令** | 用例含精确的 data-testid 定位和可程序化验证条件 |
| **确定性数据** | SQL脚本初始化固定测试数据，消除环境不确定性 |
| **双层验证** | UI交互验证 + API响应断言，确保端到端正确性 |

### 1.2 与v4.0的区别

| 维度 | v4.0 | v5.0 |
|------|------|------|
| 用例数量 | ~85个核心 + 5个E2E | ~123个核心 + 7个E2E |
| 模块覆盖 | 8个模块 | 10个模块（新增智能分配+通知） |
| 设置模块 | 5个用例 | 18个用例（覆盖全部8个Tab） |
| 导入导出 | 无 | 项目+任务的导入导出用例 |
| 时间线交互 | 1个用例 | 3个用例（标尺/工具栏/今天标记） |
| seed.sql | 基础数据 | 补充多状态任务/审批/通知/能力模型 |

---

### 1.3 与v3.0的区别

| 维度 | v3.0（旧） | v4.0（新） |
|------|-----------|-----------|
| 用例数量 | 574个预设文档 | ~85个核心 + AI实时补充 |
| 用例格式 | Markdown自由描述 | 结构化指令（含定位和验证） |
| 元素定位 | 无 | data-testid 映射表 |
| 测试数据 | 动态查询数据库 | SQL脚本确定性初始化 |
| 验证方式 | "显示正确的数据"（人眼） | API断言 + UI文本匹配 + 元素存在性 |
| 维护成本 | 需求变更需批量更新文档 | AI自适应UI变化，仅需更新核心用例 |

## 二、测试架构

### 2.1 执行引擎

**Chrome DevTools MCP** — AI直接操作浏览器

优势：
- 无需额外测试框架
- 支持多Browser Context并发
- 支持Lighthouse性能审计
- 支持性能追踪（Performance Trace）
- AI可通过take_snapshot实时探索页面结构

### 2.2 用例层级

| 层级 | 目的 | 用例数 | 文档位置 |
|------|------|--------|---------|
| **L1 核心路径** | 验证关键功能正确性 | ~123 | test-cases/core/ |
| **L2 关键业务流** | 端到端业务场景 | ~7 | test-cases/e2e/ |
| **L3 AI探索** | AI实时生成边界用例 | 动态 | 执行时生成 |

### 2.3 验证策略

| 验证类型 | 方法 | 示例 |
|---------|------|------|
| URL验证 | 检查当前URL | url contains /dashboard |
| 元素存在 | data-testid查找 | [data-testid=xxx] exists |
| 文本匹配 | 元素文本内容 | text contains "登录成功" |
| API断言 | 检查API响应 | GET /api/auth/me → role === "admin" |
| 元素数量 | 统计匹配元素 | count([data-testid=task-row]) === 5 |
| 视觉验证 | 截图对比 | 截图与基线对比 |

## 八、执行流程

```
1. 执行 seed.sql 初始化测试数据
2. AI 按模块顺序执行 L1 核心用例
   - 每个用例：take_snapshot → 执行操作 → 验证结果 → 记录
3. 执行 L2 关键业务流
4. AI 自由探索生成 L3 边界用例
5. 汇总测试报告
6. （可选）执行 seed.sql 清理测试数据
```
